# Итоговая конфигурация, производительность и рекомендация по сервису

**Объединение сработок в инциденты для Ситуатора 2.0 · для руководителя разработки**

## 1. Итоговая конфигурация

Конвейер на проверенных открытых компонентах, прогнан на полной выгрузке (4 суток, 8 692 сработки,
9 камер, 6 объектов). Вход — **только видео** (как в проде):

```
сработка (clip.mp4) → кадры из клипа (NVDEC в проде / ffmpeg) → детектор деградации сигнала
  → YOLO11x@1280 на GPU (люди/техника/животные, пер-классовые пороги, стабилизация счёта)
  → подавление статичных объектов сцены → темпоральные сессии (правило ядра)
  → межкамерная склейка по графу соседних камер → карточки + описания (словарь, без LLM)
  ↘ на ФРОНТАЛЬНЫХ камерах (ворота): плате-детектор → РФ-OCR (наша) → confidence+кратность-гейт → госномер
выход: карточка инцидента {тип, период, камеры, описание, госномер?, trigger_ids} + события ре-флага
```

- **Детекция:** YOLO11x@1280 (не дешёвая yolo11s) — ловит мелкое/ночное, что лёгкая модель теряет
  (ночной человек на тёмной камере: yolo11s не увидел, yolo11x нашёл person 0.34).
- **Номера:** собственная РФ-OCR (дообученный fast-plate-ocr, CCT-S v2, кириллица), held-out
  char-acc 99.99%; глобальная OCR путала У→М/Н→В — наша чинит. Только на фронтальных камерах.
- **Безопасность:** пол «0 потерь REAL» — поштучная проверка на каждом прогоне (не «by construction»).

## 2. Результаты против исходной автоматики

| Метрика | Ситуатор 2.0 | Наш конвейер |
|---|---|---|
| Новых карточек-инцидентов на разбор | 3 762 (≈39/час) | **801 (≈8.4/час)** — в 4.7× меньше |
| + ре-флаги эскалации (обновления карточек) | нет | **558** |
| **Итого операторских событий (РЕАЛЬНАЯ нагрузка)** | **3 762 (≈39/час)** | **1 359 (≈14/час)** — **в 2.8× меньше** |
| Сжатие потока | 1.42× (видимая часть) | **8.6×** (8 692→1 011 карточек) |
| Межкамерных инцидентов | 0 | **61** |
| Потеряно реальных событий (REAL) | — | **0 из 178** |
| Госномера (фронтальные камеры) | нет | точные РФ, доверяемые (гейтинг) |

**Честно о нагрузке.** Считать только карточки неправильно: ре-флаг — это тоже событие оператору.
Реальная нагрузка = карточки + ре-флаги = **1 359 событий ≈ 14/час против 39/час** (в 2.8× меньше).
Из них новых инцидентов на разбор — 801 (в 4.7× меньше), остальные 558 — **лёгкие обновления-
эскалации** (карточка уже на экране, оператор видит «появился человек / +камера»), и это **новая
способность**, которой у Ситуатора нет вовсе: вместо похороненной эскалации — видимое обновление.
Плюс 0 потерянных REAL и распознавание номеров.

## 3. Производительность и ёмкость (вопрос про 200+ камер)

Замер — нативно на Apple M5 (MPS), полный прогон 8 692 сработок:

| Показатель | Значение (8 692 сработки, 25 мин) |
|---|---|
| Пик RAM | **1 874 МБ** |
| Пик VRAM (GPU) | **1 184 МБ** (5% от 24 ГБ) |
| CPU (эквивалент ядер, из 18) | **0.2 средн. / 2.0 пик** |
| YOLO11x@1280 на активную сработку | **261 мс** (p95 314) |
| Пропускная способность узла (последовательно) | **3.8 активных сраб/с** |

**Расчёт на прод.** Нагрузка определяется **темпом сработок**, не числом камер: сработка — это
событие детектора движения, а не непрерывный поток. На контуре 9 камер пик — 0.75 сработки/с
(≈0.08/с на «горячую» камеру). Линейная экстраполяция на 200 камер: пик ≈ **15–18 сработок/с**.

- На целевой 3090 (≈×1.6 к M5-MPS) один узел тянет ≈ **6–7 активных сработок/с** при YOLO11x@1280 →
  на 200 камер **≈ 3 GPU-узла** под детекцию **последовательно** (консервативно).
- **Это нижняя граница.** Замер последовательный: VRAM занят на 5%, CPU на 0.2 ядра из 18 — огромный
  недогруз. **Батчинг GPU** (десятки кадров за вызов) реально даёт ×3–10 → вероятно **1 узел** на
  200 камер. Батчинг не меняет результаты (инференс детерминирован покадрово) — см. §3а.
- VRAM один узел: 1.2 ГБ из 24 → помещается ещё ALPR ~2-3 ГБ и VLM-7B ~8 ГБ с запасом.
- Декод видео в проде — **NVDEC** (не CPU-ffmpeg как на Mac): сработки — короткие клипы
  (event-driven декод), нагрузка кратно ниже непрерывного потока.
- Стрим-логика (сессии, склейка, ре-флаг, гейтинг номеров) — **копейки CPU**, не масштабная статья.

**Вывод: 200+ камер реалистичны** на единицах GPU-узлов (≈3 последовательно, ≈1 с батчингом);
точное число — по фактическому темпу сработок контура (замерить на их данных).

## 3а. Готовность к масштабированию (scaling-readiness) — честно

- **Вертикально (батчинг на одном GPU) — готовы, без дрейфа.** YOLO детерминирован покадрово, батч
  лишь группирует → результаты идентичны. Главный бесплатный выигрыш.
- **Горизонтально (по узлам) — по логике готовы, по инфраструктуре нет.** Без-состояния стадии
  (кадры, деградация, YOLO, OCR) масштабируются тривиально. Состояние-зависимые (сессии, склейка,
  ре-флаг) — **партиционированием по ОБЪЕКТУ** (соседство всегда внутри объекта → склейка локальна
  воркеру) с сохранением порядка внутри шарда. Логику переписывать не нужно.
- **Одна замена под капотом — статик-карта** (сейчас глобальный батч-проход → нужна скользящая
  причинная). **Дрейф метрик возникнет при наивном параллелизме** (порядок не сохранён / карта
  неконсистентна / шард по камере). При корректном (объект + порядок + скользящая карта) результаты
  **точно совпадают** с однопоточной, пол «0 потерь REAL» держится. Итог: batch-ready сразу;
  partition-ready по логике, нужна достройка стриминг-приёма (роадмап) + замена статик-карты.

## 4. Принятые технические решения

- **YOLO11x@1280 на GPU вместо лёгкой модели на CPU.** Замер: с дешёвой моделью CPU успевал, но
  ночь/мелочь терялись; точная модель требует GPU (на CPU 0.49 сраб/с — ниже пика). GPU оплачивает качество.
- **Стабилизация счёта объектов** — не Кальман: пер-сработочный пик + гистерезис (полосы 2/4/8/16),
  ре-флаг по входу в новую полосу, иначе спам. (Истинный трекинг с ID — на потом, под Re-ID.)
- **Ре-флаг — событие, не тихое дорастание карточки.** При эскалации (появился человек / +объекты /
  подключилась камера) карточка возвращается оператору как UPDATE. Решает риск «отработал стартовые —
  такими и остались».
- **Номера: собственная РФ-модель + гейтинг от галлюцинаций.** Confidence (min char_prob ≥0.80) +
  кратность (≥2 чтений) обвалили 55 фантомных строк до 5 доверяемых точных номеров на воротах.
- **Пол безопасности — эмпирический per-run аудит**, не доказательство по построению (REAL с
  пропущенной детекцией структурно не защищён ни в батче, ни в стриме — держит только проверка).

## 5. Учтённые особенности и неудачные попытки

- **macOS + Docker + GPU несовместимы.** Docker на Mac = Linux-VM, Apple-GPU (Metal) в неё не
  пробрасывается, CUDA на Mac нет. Поэтому для GPU-замеров пайплайн запускается **нативно** (venv,
  device=mps); рабочий конвейер остаётся в Docker (CPU) для воспроизводимости. На целевом железе
  (Linux+NVIDIA) Docker+`--gpus` работает штатно — там противоречия нет.
- **Что пробовали неудачно:** *nomeroff-net* (РФ-ALPR) — хрупкий форк (deps из git, слом torch.load) →
  бросили, дообучили **чистый** fast-plate-ocr; *ALPR на парковочной камере* — номера физически нечитаемы
  (разрешение) → только фронтальные камеры; *CoreML EP* падал на пустых детекциях → форс CPU;
  *пересборка docker ради правки .py* — стёрла кэш, тянула torch+CUDA ~40 мин → перешли на bind-mount.

## 6. Особенности формата данных (что появилось)

- **Карточка инцидента:** id, тип (`реал`/`несхлопнуто`/`среда`/`деградация`), период, камеры,
  описание (≤160 симв., словарь, без LLM), `trigger_ids`. `среда`/`деградация` — агрегаты камера-час.
- **Ре-флаг (новое):** события `CREATE/UPDATE/FINALIZE`; UPDATE = эскалация (причина+дельта) — для оператора это новая сработка по карточке.
- **Госномер (новое):** строка + уверенность; только фронтальные камеры, после гейтинга; для идентичности/белого списка/Re-ID транспорта.
- **Граф соседних камер — НУЖЕН** (включает межкамерную склейку; иначе 0 межкамерных). Сравнение только внутри связанных пар. Заполняется вручную по контуру — подтверждена 1 пара, остальные за заказчиком.
- **Камера — из `list.txt`** (`/recordings/camXX/`), не `sensor_name` (заполнен ~36%); **часовой пояс на объект** — для «утром/ночью».

## 7. Рекомендация: сделать сервис на базе нашего репозитория

Прототип — не презентация, а **работающий код в репозитории**, который запускается тремя командами
и воспроизводит эти числа. Рекомендуем оформить его в **сервис рядом с Ситуатором 2.0**: на вход —
поток сработок (как из Интеллекта: id + клип), на выход — карточки инцидентов + события ре-флага +
номера через API/очередь.

Первоочередное (необходимое условие боя): **стриминговый API** — приём сработок, открытые сессии в
памяти, публикация карточки/ре-флага наружу; батч-логика уже эквивалентна реплею потока. Дальше по
ценности: классификатор «реал/шум» под камеру, VLM-арбитр на кандидатах (события-состояния: ворота
открыты — YOLO слепа, VLM в 24 ГБ видит), граф соседних камер по контуру, UI оператора.

Репозиторий: **https://github.com/SergeiKarulin/situator-incident-pipeline** (обезличен; данные и
имена объектов заказчика не публикуются). Этот отчёт — там же.

## Приложение. Как работают ре-флаги — на конкретном случае

Ре-флаг — событие `UPDATE` на уже созданной карточке: когда обстановка усиливается (появился
человек, выросло число объектов, подключилась соседняя камера), карточка возвращается оператору
наверх очереди с пометкой причины. Эскалация не хоронится: «одна машина» на глазах превращается в
«группа вошла», и оператор это видит, а не закрывает карточку по стартовому состоянию.

**Случай: машина заехала → собралась группа и вошла** (реальный журнал прогона, имена обезличены):

| Время | Событие | Что увидел оператор |
|---|---|---|
| 04:01 | CREATE | камера двора: машина (1 сработка) |
| 04:02 | UPDATE · рост числа машин (≥2) | подъехала вторая |
| 04:02 | UPDATE · **появился человек** | ← ключевая тревога |
| 04:02 | UPDATE · рост числа людей (≥2) | людей больше |
| 04:03 | UPDATE · **подключилась камера ворот** | межкамерный охват |
| 04:13 | UPDATE · новый объект: собака, затем грузовик | состав изменился |
| 04:18 | UPDATE · рост числа людей (≥5) | группа |
| 04:48 | FINALIZE | один инцидент: люди + собака + машины + грузовик, 2 камеры, 263 сработки |

Без ре-флагов оператор увидел бы стартовую «машина во дворе», закрыл как рутину, а вход группы был
бы похоронен внутри той же карточки. С ре-флагами каждое усиление — отдельное событие (8 на этот
инцидент вместо 1; учтено в «реальной нагрузке» §2); зато эскалация видна вживую. **Типы причин:**
*появился человек* (высшая ценность), *подключилась соседняя камера* (межкамерный охват),
*новый объект*, *рост числа* (полосы 2/4/8/16). За 4 суток — 558 ре-флагов, ~⅔ содержательные
(человек/камера/объект); шумные «+1 на тик» отсекаются гистерезисом и полосами.
