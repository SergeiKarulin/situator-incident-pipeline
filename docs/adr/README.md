# Журнал архитектурных решений (ADR)

**ADR** (Architecture Decision Records) — лог существенных архитектурных и методических решений: что
выбрали, на какие факты опирались, что отвергли и какие последствия это влечёт. Цель — чтобы любой
(человек или новая LLM-сессия) понимал, *почему* система устроена так, и не переоткрывал уже взвешенные
развилки.

Внутренний документ — имена камер/объектов и специфика допустимы; **в публичный репозиторий не идёт**
(там обезличенный отчёт). Записи пронумерованы по хронологии принятия. Дата журнала: 2026-06-19.

---

## ADR-001. Локальный деплой, не аренда облачного GPU
- **Статус:** принято
- **Контекст и факты:** данные — видео с частных охраняемых объектов (PII, режимность). У заказчика
  есть локальная машина (2× RTX 3090, 24 ГБ). Замер показал огромный недогруз одного GPU (VRAM ~5%,
  CPU 0.2 ядра из 18) — мощности локального железа с запасом.
- **Решение:** весь конвейер — локально, offline, на открытых компонентах; без отправки данных вовне.
- **Альтернативы:** аренда A10/облака — отвергнута: вынос чувствительных данных наружу + постоянная
  оплата при простаивающем железе у заказчика.
- **Последствия:** ноль внешних зависимостей и утечек; стек ограничен open-source; масштабирование —
  своими узлами (см. ADR-019 про ёмкость).

## ADR-002. Докеризация пайплайна; GPU-прогоны — нативно на MPS
- **Статус:** принято
- **Контекст и факты:** требование «всё в докеры, никакого мусора». Но Docker на Mac работает в
  Linux-VM **без проброса Metal** — GPU (Apple MPS) из контейнера недоступен. Прогон YOLO11x на CPU в
  контейнере давал 0.49 сраб/с — ниже пикового темпа контура (0.75 сраб/с), т.е. не успевал.
- **Решение:** прод-конвейер докеризован (воспроизводимость, чистота); тяжёлые GPU-замеры/прогоны —
  нативно в `.venv-mps` на MPS. В проде (Linux + NVIDIA) Docker видит GPU штатно, расхождения нет.
- **Альтернативы:** только Docker на Mac — отвергнуто (нет GPU → не успевает в реальном времени).
- **Последствия:** два пути запуска на деве; нативный путь (`tools/bench_full.py`) — источник
  замеров производительности и качества.

## ADR-003. YOLO11x@1280 на GPU вместо лёгкой yolo11s на CPU
- **Статус:** принято
- **Контекст и факты:** на тёмной ночной камере yolo11s **не увидел** реального человека; yolo11x@1280
  нашёл его (person 0.34). Лёгкая модель на CPU успевала по скорости, но теряла мелкое/ночное.
- **Решение:** детектор — YOLO11x@1280 на GPU.
- **Альтернативы:** yolo11s@960 на CPU — отвергнуто: цена скорости — потеря REAL (нарушение пола, ADR-012).
- **Последствия:** нужен GPU (ADR-001); ~260 мс/активную сработку; recall ночью выше.

## ADR-004. Вход — только видео, jpg игнорируем
- **Статус:** принято
- **Контекст и факты:** в проде сработка приходит как `clip.mp4`; jpg-превью в выгрузках сгенерированы
  для удобства анализа и в бою отсутствуют. Подтверждено заказчиком.
- **Решение:** конвейер работает от видео (кадры извлекаются из клипа); jpg при индексации не нужны.
- **Альтернативы:** опора на jpg — отвергнуто: их нет в проде, путь стал бы нерабочим.
- **Последствия:** нужен декод (NVDEC в проде / ffmpeg на деве); выбор кадров — наша забота (ADR-015).

## ADR-005. Пер-классовые пороги уверенности детектора
- **Статус:** принято
- **Контекст и факты:** один глобальный conf терял события. Калибровка (Experiment_004): лесной REAL
  давал person ~0.24 → порог person опущен до 0.20; boat НЕЛЬЗЯ поднимать выше 0.45 — на 0.65 теряются
  REAL, где техника под чехлом детектится как boat (вертолёт/лодка). Животные ~0.30.
- **Решение:** `conf_per_class` (person 0.20, default 0.45, животные 0.30, boat 0.45-не-выше).
- **Альтернативы:** единый порог — отвергнут (либо шум, либо потери REAL).
- **Последствия:** цена низкого порога — единичные шумовые карточки; их добивают другие ступени.

## ADR-006. Правило «подтверждённого ядра» (реал / несхлопнуто / среда)
- **Статус:** принято
- **Контекст и факты:** 2 случайных блоба за 2.5 ч дождя не должны клеиться в «реал»; минивэн с
  регулярными хитами — должен оставаться одним инцидентом.
- **Решение:** активные сработки группируются в «ядра» тем же tau_gap; ядро подтверждено, если в нём
  **≥2 активных сработки ЛИБО объект виден в ≥2 кадрах** одной. Подтверждённое ядро → «реал»; одиночная
  слабая улика → «несхлопнуто» (оператору, не теряем); прочий шум → «среда» (авто-закрыто).
- **Альтернативы:** «любая детекция = инцидент» — отвергнуто (шум); «жёсткий порог по числу» —
  отвергнуто (теряет одиночные REAL → их спасает «несхлопнуто»).
- **Последствия:** промежуточный тип «несхлопнуто» как страховка recall; повышение метки только вверх.

## ADR-007. Ре-флаг по эскалации (полосы 2/4/8/16 + гистерезис)
- **Статус:** принято
- **Контекст и факты:** сценарий заказчика — одиночная сработка, перерастающая в группу, НЕ должна
  оставаться похороненной в стартовом состоянии. Наивный ре-флаг давал спам (25× «+1 person»).
- **Решение:** карточка возвращается оператору (UPDATE) при входе в новую полосу счёта (2/4/8/16),
  появлении человека, подключении камеры, новом объекте; дебаунс up/down гасит дребезг.
- **Альтернативы:** ре-флаг на каждое +1 — отвергнут (спам); без ре-флага (как у Ситуатора) —
  эскалация хоронится внутри группы.
- **Последствия:** ре-флаг — тоже операторская реакция, учитывается в нагрузке; уникальная способность
  против Ситуатора.

## ADR-008. Пер-камерная карта статичной «мебели»
- **Статус:** принято
- **Контекст и факты:** в кадре каждой камеры постоянно стоят объекты (припаркованная машина, лодка под
  чехлом, ряд авто) — они штормят детектор движения, но инцидентом не являются.
- **Решение:** объект, висящий в ≥35% сработок камеры дольше ~4 ч, считается фоном **этой камеры** и
  подавляется (карта строится `by_camera`, автоматически из данных). Людей/животных не подавляем.
- **Альтернативы:** глобальное подавление — отвергнуто (сцены камер различны).
- **Последствия:** в стриме карта пока берётся предрассчитанной (батч); причинный rolling-map отложен
  (см. ADR-019, scaling).

## ADR-009. Межкамерная склейка по графу соседства; граф должен быть ПОДТВЕРЖДЁН
- **Статус:** пересмотрено (уточнено vision-проверкой)
- **Контекст и факты:** проверка зрением 10 межкамерных случаев: на ПОДТВЕРЖДЁННОЙ паре (смежные
  гаражные боксы) ре-флаг «подключилась камера» осмыслен (человек реально ходит между боксами); на
  УГАДАННОЙ паре двор↔ворота — в основном ложный (ворота пусты / разные объекты). В Эксп.2 все 26 наших
  межкамерных — на этой спорной паре; у Ситуатора 8 — там же.
- **Решение:** межкамерная склейка только внутри пар графа соседства; **граф брать подтверждённым у
  эксплуатации (у Ситуатора он есть), а не угадывать**. Неподтверждённые пары — отключать.
- **Альтернативы:** автосклейка по близости времени без графа — отвергнута (ложные склейки).
- **Последствия:** качество межкамерного = качество графа; до подачи реального графа межкамерное у
  обеих сторон спорно.

## ADR-010. ALPR только на фронтальных камерах + собственная РФ-OCR
- **Статус:** принято
- **Контекст и факты:** на воротах (камера ворот) номер читается глазами (`‹номер›`); на парковке (парковочная камера)
  нечитаем даже человеком (разрешение не несёт символы). Глобальная OCR путала кириллицу (У→М, Н→В,
  теряла регион).
- **Решение:** распознавание номеров только на фронтальных камерах; OCR — собственная РФ-модель
  (дообученный fast-plate-ocr CCT-S v2, кириллица), held-out char-acc 99.99%; гейт min char_prob≥0.80 +
  кратность≥2 против фантомов. На камера ворот: 55 сырых → 5 доверяемых точных номеров.
- **Альтернативы:** глобальная OCR — отвергнута (кириллица); nomeroff-net — отвергнут (версионно-хрупкий,
  dep-hell). ALPR на всех камерах — бессмысленно (нечитаемо).
- **Последствия:** номера — только «свой/чужой» на воротах; на новой выгрузке (Эксп.2) доверяемых 0
  (за период не было читаемых въездов).

## ADR-011. Дообучение РФ-OCR изолированно; данные заказчика не идут в обучение/апстрим
- **Статус:** принято
- **Контекст и факты:** требование жёсткой гигиены данных: номера/кадры заказчика ни в каком виде не
  должны попасть в открытый репозиторий/апстрим fast-plate-ocr.
- **Решение:** дообучение вынесено в отдельную папку-сосед/инстанс; обучение — только на публичных
  данных (Nomeroff RU 57k); для демо/проверки — картинки из интернета. Acceptance-кропы заказчика —
  локально, не публикуются.
- **Альтернативы:** обучать на данных заказчика — запрещено.
- **Последствия:** возможный PR авторам fast-plate-ocr — из чистого экспорта, без первички.

## ADR-012. Пол «0 потерь REAL» — per-run аудит, не «by construction»
- **Статус:** принято
- **Контекст и факты:** структурной гарантии «REAL никогда не уйдёт в авто-закрытие» нет (REAL с
  пропущенной детекцией структурно не защищён). На блоке 1–4 июня поштучная проверка дала 0/178 потерь;
  на 17–19 — 1/37 (см. ADR-015).
- **Решение:** пол держится **поштучной проверкой на каждом прогоне**, а не объявляется по построению.
- **Альтернативы:** заявлять «by construction» — отвергнуто (неправда, замаскировала бы потери).
- **Последствия:** каждый прогон требует REAL-аудита; честно сообщаем найденные потери.

## ADR-013. Консервативная недосклейка вместо пересклейки
- **Статус:** принято
- **Контекст и факты:** пересклейка ХОРОНИТ событие (новый инцидент исчезает внутри старой карточки) —
  необратимо для оператора; недосклейка лишь даёт лишнюю карточку (поправимо). Vision подтвердил, что
  крупная наша карточка (212 сраб, 34 мин активности во дворе) — связное событие, не пересклейка.
- **Решение:** при сомнении — НЕ склеивать; любые авто-объединения через ключ/подтверждение, не молча.
- **Альтернативы:** агрессивная склейка ради сжатия — отвергнута (риск похоронить REAL).
- **Последствия:** сжатие чуть ниже теоретического максимума, зато без потерь событий.

## ADR-014. LLM/VLM — не вратарь; идентичность из ключей, не из LLM
- **Статус:** принято (роадмап)
- **Контекст и факты:** штурм тремя субагентами сошёлся: «человек/собака» бесполезно (дежурный пишет
  идентичность — Иван Иваныч/горничная). Имя один раз вводит человек, а детерминированный ключ
  (номер/Re-ID) его помнит и подставляет — это match/lookup, не LLM.
- **Решение:** LLM/VLM повышает информативность и сигналит на проверку, но НЕ авто-действует и не
  закрывает молча (человек в петле); пол «0 потерь REAL» — на CV. Идентичность — из ключей + меток
  оператора. Подробности — `Идеи_использования_LLM_TBD.md`.
- **Альтернативы:** VLM-аннотации «что на картинке» как основной кейс — отвергнуто (низкая ценность).
- **Последствия:** приоритет улучшений — пер-камерный классификатор (реакции) + видео-гейтинг (потери),
  а не VLM; VLM — узкая ниша по кнопке (состояния/поведение).

## ADR-015. Видео-гейтинг: motion-guided кадры вместо 4 фиксированных
- **Статус:** пересмотрено(→ADR-021)
- **Контекст и факты:** 4 кадра на фикс. секундах слепнут к движению между ними. Единственная потеря в
  Эксп.2 (`6369614`) — перекрытый прутьями ворот, далёкий, почти неподвижный человек: YOLO даёт 0
  детекций даже при conf 0.10, motion = только листва (0.6–2.7%). Motion-detect (absdiff) на 2 порядка
  дешевле YOLO@1280.
- **Решение:** дешёвый motion по плотным кадрам → YOLO на **2 кадрах на пиках движения** (≥2 сохраняет
  правило ADR-006), 1 кадр-фолбэк при отсутствии движения; страховка: локализованное движение есть, но
  YOLO молчит → «несхлопнуто» (видимо), не «среда».
- **Альтернативы:** 1 кадр — отвергнуто (теряется подтверждение ≥2 кадров, хрупкость к смазу). 4 фикс. —
  избыточно при наличии motion.
- **Последствия:** меньше вызовов YOLO (главная статья) + точнее. ВАЖНО: конкретную потерю `6369614`
  видео-гейтинг НЕ вернёт (объект невидим детектору) — это предел детектора на перекрытой/далёкой цели,
  отдельная задача (детектор ROI ворот).

## ADR-016. Пер-камерная калибровка — кросс-блочно (без утечки)
- **Статус:** принято
- **Контекст и факты:** подгонка порогов под тот же датасет, на котором меряешь, — самообман. Есть два
  независимых блока данных (1–4 и 17–19 июня).
- **Решение:** пер-камерные/дневные-ночные пороги калибровать на одном блоке, проверять на другом.
- **Альтернативы:** калибровка и замер на одном блоке — отвергнуто (data leakage).
- **Последствия:** честная оценка переноса; нужен ≥1 блок «на обучение».

## ADR-017. Глобальные пороги первыми, пер-камерные — потом
- **Статус:** принято
- **Контекст и факты:** холодный старт — пер-камерному классификатору нужны размеченные «реал/шум» НА
  КАЖДУЮ камеру, а REAL редкий (~2%, единицы на камеру за сутки). Происхождение меток в xlsx неизвестно.
  В Эксп.2 появились новые камеры (новая камера/новая камера) без истории — глобальный порог работает на них сразу.
- **Решение:** базовый слой — глобальные пороги + правило ядра; пер-камерное — следующий шаг при
  накоплении истории и с recall-гардом (нельзя задавить редкий реал на камере).
- **Альтернативы:** сразу пер-камерные модели — отвергнуто (не на чем учить, нет фолбэка для новых камер).
- **Последствия:** глобальный слой — обязательный фолбэк навсегда; пер-камерное — улучшение поверх.

## ADR-018. Анонимизация отчёта в публичном репозитории
- **Статус:** принято
- **Контекст и факты:** репозиторий публичный (для разработчиков). Имена объектов, ID камер, реальные
  номера — чувствительны.
- **Решение:** в репо идёт обезличенный отчёт (камеры → роли «двор/ворота/площадка», без объектов и
  реальных номеров); проверка grep перед каждым пушем; первичка в репо не коммитится. ADR-журнал и
  сырые данные — только локально.
- **Альтернативы:** публиковать как есть — отвергнуто (утечка данных заказчика).
- **Последствия:** двойной комплект: внутренний (полный) и публичный (обезличенный).

## ADR-019. Два эксперимента; сравнение с самоулучшившимся Ситуатором
- **Статус:** принято
- **Контекст и факты:** между выгрузками программисты докрутили Ситуатор: межкамерное 0→8 (структурное
  изменение — из «ровно 0» так не выйдет без смены логики), сжатие 1.69→4.86×. Замер на одной выгрузке
  (равные условия): мы 14 реакций/ч против их 19/ч, сжатие 10.98× против 4.86×.
- **Решение:** мерить себя против ТЕКУЩЕГО (улучшенного) Ситуатора, на одной выгрузке; отчёт разбит на
  Эксперимент 1 (1–4 июня) и Эксперимент 2 (17–19 июня).
- **Альтернативы:** сравнивать с исходным Ситуатором — отвергнуто (нечестно завышает наш отрыв).
- **Последствия:** порог сравнения выше; наш отрыв реальный, но ýже, чем был; решение о внедрении — с
  условиями (видео-гейтинг + пер-камерное + реальный граф соседства).

## ADR-020. Темпоральные сессии (tau_gap) + шторма деградации
- **Статус:** принято
- **Контекст и факты:** сработки — событийный поток одной (объект, камера); серия с паузами ≤ tau_gap —
  один эпизод. Часы шумящей по сигналу камеры (шумная камера) дают сотни «деградаций».
- **Решение:** сессия = серия одной (объект, камера) с паузами ≤ tau_gap (150 с, пер-объектный override
  поддержан); час камеры с ≥ storm_min деградированных изымается в одну карточку «деградация».
- **Альтернативы:** фикс. окна — отвергнуто (рвут эпизоды / склеивают разное).
- **Последствия:** деградация не засоряет обычный поток; tau_gap — параметр под докалибровку.

## ADR-021. Видео-гейтинг реализован как motion-страховка; конфигурируемый, по умолчанию OFF (пересмотр ADR-015)
- **Статус:** принято
- **Контекст и факты:** реализацию ADR-015 проверили на ДВУХ боевых выгрузках; замеры:
  - Новый блок (17–19 июня, **7061 сработка**): метрики с motion-страховкой = ровно baseline — **643
    карточки, 471 видимых, потеря REAL 1** (`6369614`). Единственная потеря — неподвижный, перекрытый
    прутьями ворот человек: YOLO даёт **0 детекций даже при conf=0.10 на 48 плотных кадрах**, motion-
    сигнала нет (только листва, **0.6–2.7%**). Страховка её НЕ возвращает.
  - Старый блок (1–4 июня): baseline и так **0 потерь из 178** → страховке возвращать нечего (польза
    невозможна по построению). Не гоняли ради предсказуемого нуля.
  - Попытка motion-guided СОКРАЩЕНИЯ кадров (1–2 вместо 4, как предлагал ADR-015) РЕГРЕССИРОВАЛА recall:
    **8 потерь REAL вместо 1** (+ баг покрытия: декодировались только первые 6 с клипа). Откатили к
    проверенным **4 фиксированным кадрам**; motion оставлен только отдельным сигналом для страховки.
  - Калибровка порога движения: строгий порог (устойчивость ≥2 пар кадров, блоб ≥0.3% кадра) убрал
    флуд FALSE с **35% до 1.7%**, но REAL-движение ловит лишь **37%** (большинство REAL берёт YOLO
    напрямую).
- **Решение:** motion-страховка («есть устойчивое локализованное движение, но YOLO молчит → показать
  оператору, не хоронить в среду») реализована, но **вынесена в конфиг как отключаемая, по умолчанию
  OFF**: `pipeline/config/pipeline.yaml` → `video_gating.enabled: false`; override `env USE_MOTION=1`.
  Выбор кадров — 4 фиксированных (НЕ motion-guided сокращение, см. факт о регрессии).
- **Альтернативы:** оставить включённой по умолчанию — отвергнуто (нет доказанной пользы на наших
  данных, риск флуда FALSE). Удалить совсем — отвергнуто (на бо́льшем объёме польза возможна; оставляем
  отключаемой). motion-guided СОКРАЩЕНИЕ кадров — отвергнуто (8 потерь REAL вместо 1).
- **Последствия:** на наших данных motion-страховка НЕЙТРАЛЬНА — recall-safe, но пользы 0. Ценность
  гипотетическая: робастность к будущим стоп-кадровым промахам ДЕТЕКТИРУЕМЫХ движущихся объектов;
  включать только после валидации на бо́льшем объёме. Текущую единственную потерю (`6369614`) закрывает
  не motion, а отдельный детектор ROI ворот (occlusion/высокое разрешение) — см. ADR-015.

## ADR-022. Объединение-продолжение по жёсткому ключу (continuation-merge); конфигурируемо, по умолчанию OFF
- **Статус:** принято
- **Контекст и факты:**
  - Концептуальное разграничение с ре-флагом (ADR-007): ре-флаг работает на **ОТКРЫТОМ** инциденте по
    ключу (object, camera) + временна́я непрерывность (паузы < τ_gap) — личность субъекта не нужна.
    Продолжение — это **переоткрытие ЗАКРЫТОГО** инцидента через разрыв > τ_gap, где временна́я связка
    разорвана → нужен жёсткий ключ идентичности (госномер/Re-ID), иначе склейка по времени хоронит
    события (см. ADR-013).
  - Замер на старом блоке (1–4 июня), ворота камера ворот, **146 сработок**: распознано **5 доверяемых
    номеров** (‹номер›, ‹номер›, ‹номер›, ‹номер›, ‹номер›) — КАЖДЫЙ появился **РОВНО ОДИН
    раз**, возвратов через > τ_gap = **0**. На новом блоке (17–19 июня) номеров распознано **0**. Re-ID
    не реализован.
  - Итог: возможностей склейки на наших данных = **0**. Метод здравый, но не на чем проявиться (ключи
    редкие + короткое окно без возвратов).
- **Решение:** реализована функция склейки `pipeline/situator/continuation.py` (`merge_continuations`:
  закрытый инцидент + возврат ТОЙ ЖЕ сущности по жёсткому ключу — госномер/Re-ID — к тому же объекту в
  пределах окна → продолжение, а не новая карточка). Вынесена в конфиг
  `continuation_merge.enabled: false` (по умолчанию OFF), `window_minutes=60`, `key=plate|reid`. Есть
  синтетический selftest (проходит).
- **Альтернативы:** склейка по (object, camera, время) без жёсткого ключа — отвергнута (это раздувание
  τ_gap → пересклейка, хоронит события, см. ADR-013); удалить совсем — отвергнуто (потенциал реальный:
  на охраняемом объекте машины жильцов/персонала возвращаются постоянно).
- **Последствия:** OFF по умолчанию; включать при плотном источнике ключей (Re-ID или плотнее читаемые
  номера). Полная интеграция в стрим-движок (плате→инцидент) отложена до появления плотных ключей —
  сейчас 0 эффекта, не вносим неэкзерсируемую сложность. В руководство по внедрению НЕ выносим (пользы
  на данных нет).

## ADR-023. Re-flag уже ОБРАБОТАННЫХ оператором инцидентов — откладываем до интеграции с боевым Ситуатором
- **Статус:** отложено
- **Контекст и факты:**
  - Ре-флаг сейчас работает ТОЛЬКО на ОТКРЫТОМ инциденте: серия сработок с паузами < τ_gap, связка по
    ключу (object, camera) + временна́я непрерывность, без идентичности субъекта (см. ADR-007).
  - Есть ДВА разных «закрытия» инцидента: (1) наш алгоритмический FINALIZE по τ_gap — это состояние
    знает сам конвейер; (2) обработка/закрытие карточки ОПЕРАТОРОМ — это состояние живёт в Ситуаторе
    (его UI/БД), у автономного конвейера («голый комбайн», без живого Ситуатора) его НЕТ. Мы не знаем,
    какой инцидент оператор уже разобрал, и не умеем дописывать update в его хранилище инцидентов.
  - Сценарий частый: оператор разобрал карточку → активность возобновилась → в идеале поднять ту же
    карточку, а не плодить новую.
- **Решение:** на автономном конвейере НЕ прорабатываем re-flag уже закрытых/обработанных оператором
  инцидентов. Откладываем до интеграции с боевым Ситуатором.
- **Почему откладываем:** прорабатывать re-flag обработанных инцидентов в отрыве от реального Ситуатора —
  работа вслепую: проектирование против неизвестного интерфейса и неизвестной модели состояния обработки.
  Это интеграционная задача (читать ID + статус обработки инцидента из Ситуатора, писать обновление
  назад), а не алгоритм нашего конвейера.
- **Альтернативы:** реализовать на нашей модели «закрытия по τ_gap» — отвергнуто (τ_gap-закрытие ≠
  операторское закрытие, получим неверное поведение); игнорировать вовсе — отвергнуто, зафиксировали как
  задачу на интеграцию.
- **Последствия:** в руководстве по внедрению — сноска «Требует проработки при интеграции: re-flag уже
  обработанных инцидентов». Решать при интеграции с боевым Ситуатором.
